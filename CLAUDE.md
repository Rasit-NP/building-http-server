# building-http-server — 작업 문서화 지침

이 저장소는 매일의 작업과 트러블슈팅을 `docs/` 에 기록한다.
아래 규칙을 따라 **일관된 형식**으로 문서를 작성한다.

> 참고: 상위 폴더(`../CLAUDE.md`)에 "신종혁 커밋 분석" 지침이 따로 있으나, **이 저장소의 작업과는 무관**하다. 이 프로젝트의 작업 문서화는 본 파일을 우선한다.

---

## 1. 파일 위치 · 네이밍

- 모든 작업 로그는 `docs/` 에 둔다.
- 파일명: **`YYYY-MM-DD-<topic>.md`** (예: `2026-06-02-socket-refactoring.md`)
- 하루에 주제가 여럿이면 **주제별로 파일을 나눈다** (한 파일에 몰지 않는다).
- 양식은 `docs/_TEMPLATE.md` 를 복사해서 채운다.

## 2. 트러블슈팅 항목 스키마 (필수)

각 트러블슈팅 건은 반드시 아래 4필드로 기록한다. 순서 고정:

1. **증상** — 무엇이 어떻게 실패했는가 (에러 메시지/현상)
2. **원인** — 근본 원인
3. **해결** — 적용한 수정
4. **교훈** — 재발 방지를 위한 일반화 (선택, 의미 있을 때만)

## 3. 심각도 범례 (통일)

- 🔴 **빌드/실행 차단** — 고치지 않으면 컴파일·링크·실행 불가
- 🟡 **권장** — 동작하지만 견고성/품질 개선 필요
- ⚪ **향후** — 다음 단계 과제 (지금 범위 아님)

## 4. 근거 기반 원칙 (필수)

- **추측 금지.** 실제로 확인한 사실만 기록한다.
- 빌드/테스트를 **실제로 실행하고 그 명령과 결과를 문서에 첨부**한다.
  (예: `gcc:12` 컨테이너 빌드 로그, `ctest` 출력 등)
- "아마", "대략" 식 서술 대신 정량·구체적 근거를 쓴다.

## 5. 언어 규칙

- 본문은 **한국어**로 서술한다.
- **기술 용어·타입·헤더·클래스·함수명은 영어 원문**을 유지한다.
  (예: `ssize_t`, `<sys/types.h>`, `Socket::accept()`)

## 6. 필수 / 권장 섹션

- **필수**: `트러블슈팅`(2번 스키마), `빌드 · 검증`(4번 근거)
- **권장**: `작업 개요`, `남은 과제`, `핵심 교훈 요약`

## 7. 인덱스 유지

- 새 문서를 추가하면 `docs/README.md` 에 **한 줄** 추가한다:
  `- [YYYY-MM-DD] <topic> — 한 줄 요약 (파일 링크)`
- 최신 항목이 위로 오도록 정렬한다.

## 8. 커밋 컨벤션 (필수)

[Conventional Commits](https://www.conventionalcommits.org/) 형식을 따른다.
제목·본문은 **한국어**로 쓰되, 기술 용어·타입·함수명은 **영어 원문**을 유지한다 (5번 규칙 동일).

### 8-1. 제목

- 형식: **`<type>: <한국어 요약>`** (50자 내외, 마침표 없음)
- type 종류:
  - `feat` — 기능/동작 추가·변경
  - `fix` — 버그 수정
  - `test` — 테스트 추가·변경
  - `docs` — 작업 로그·문서 (인덱스 포함)
  - `refactor` — 동작 변화 없는 구조 개선
- 코드(`feat`/`fix`/`test`/`refactor`)와 작업 로그(`docs`)는 **별도 커밋으로 분리**한다.
  (관례: 코드 커밋 → 해당 작업 로그 `docs` 커밋 순서)

### 8-2. 본문

- **코드 커밋**: `- ` 불릿 목록으로 **구체적 변경**을 나열한다.
  함수·타입·시그니처·상수 등 영어 원문 식별자를 명시한다.
  - 괄호에는 **변경의 의도·동작 설명만** 덧붙인다.
    (예: `- run() 매개변수를 const std::atomic<bool>&로 변경(읽기 전용 명시)`)
  - **빌드 환경·검증 명령·측정치는 코드 커밋에 넣지 않는다.**
    (예: `(Docker Ubuntu 22.04 빌드 성공, 0.12s/exit 0)` 같은 내용 금지)
    빌드/검증 결과는 아래 `docs` 커밋과 작업 로그에만 기록한다.
- **`docs` 커밋**: 제목은 `docs: <topic> 작업 로그 및 인덱스 추가`,
  본문은 **트러블슈팅 건수 + 빌드/검증 결과 + 향후 과제**를 요약한 한 단락.
  (빌드/검증 결과는 여기에 적는다.)
- 본문 줄은 72자 내외로 접고, 이어지는 줄은 2칸 들여쓴다.

### 8-3. 금지·주의

- `Co-Authored-By` 등 trailer는 **붙이지 않는다** (최근 관례).
- 추측 금지(4번). 본문의 검증 문구는 **실제 실행한 결과만** 적는다.
